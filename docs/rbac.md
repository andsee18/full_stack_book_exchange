# Справка по системе безопасности и управлению доступом

Данный документ содержит техническое описание механизмов аутентификации, авторизации и разграничения прав доступа (RBAC), реализованных в веб-приложении "Книгообмен".

---

## 1. Модель управления доступом (RBAC)

В приложении используется ролевая модель доступа, которая определяет возможности пользователей в зависимости от их статуса.

### 1.1 Роли пользователей
1.  **GUEST** (Гость): Неавторизованный посетитель. Доступен только просмотр публичного контента.
2.  **USER** (Пользователь): Зарегистрированный участник. Может управлять своими книгами и заявками на обмен.
3.  **ADMIN** (Администратор): Обладает полным доступом к управлению пользователями и контентом системы.

### 1.2 Матрица прав доступа

| Действие | Гость | Пользователь | Администратор |
| :--- | :---: | :---: | :---: |
| Просмотр каталога и книг | ✅ | ✅ | ✅ |
| Регистрация и аутентификация | ✅ | ✅ | ✅ |
| Добавление своих книг | ❌ | ✅ | ✅ |
| Редактирование своих данных | ❌ | ✅ | ✅ |
| Управление своими заявками | ❌ | ✅ | ✅ |
| Оценка завершенных обменов | ❌ | ✅ | ✅ |
| Просмотр списка пользователей | ❌ | ❌ | ✅ |
| Назначение ролей | ❌ | ❌ | ✅ |
| Удаление любого контента | ❌ | ❌ | ✅ |

### 1.3 Техническая реализация
- **Backend**: Проверка прав осуществляется на уровне API-эндпоинтов с использованием Spring Security и аннотаций контроля доступа. При попытке несанкционированного действия сервер возвращает статус `403 Forbidden`.
- **Frontend**: Интерфейс адаптивно скрывает недоступные элементы управления. Защита маршрутов реализована через компоненты-обертки `RequireAuth` и `RequireRole`.

## 2. Система аутентификации (JWT)

Безопасность сессий обеспечивается с помощью механизма двух токенов (Access и Refresh).

### 2.1 Типы токенов
- **Access Token**:
    - Используется для авторизации каждого запроса к API.
    - Имеет короткое время жизни (по умолчанию 15 минут).
    - Передается в заголовке `Authorization: Bearer`.
- **Refresh Token**:
    - Предназначен для автоматического получения нового Access Token.
    - Имеет длительный срок действия.
    - Хранится в базе данных для возможности отзыва сессии.

### 2.2 Жизненный цикл сессии
1. **Вход**: При успешном логине сервер выдает пару токенов.
2. **Работа**: Клиент использует Access Token до его истечения.
3. **Обновление**: При получении ошибки `401 Unauthorized` клиентское приложение автоматически отправляет Refresh Token для обновления сессии без участия пользователя.
4. **Выход**: При логауте Refresh Token аннулируется на сервере, а локальные данные на клиенте очищаются.

## 3. Рекомендации по проверке механизмов

1.  **Разграничение прав**: Зайдите под учетной записью обычного пользователя и попробуйте перейти в панель администратора — система должна ограничить доступ.
2.  **Работа токенов**: В консоли разработчика (вкладка Network) можно наблюдать автоматические запросы к `/api/auth/refresh` при истечении основного токена доступа.
3.  **Безопасность выхода**: После нажатия кнопки "Выход" доступ к защищенным разделам блокируется, а повторное использование старых токенов становится невозможным.
